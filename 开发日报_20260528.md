# 开发日报｜2026-05-28

## 今日概览

今日将本轮围绕“长语音稳定性、用户转写可见、回访开场入库、最新 AI 提问展示、富主题换题、文字输入模式、录音控制区压缩、用户端统计修复、移动端与桌面端标题区统一”等一系列工程改动统一归档。

这批工作实际是一次围绕老人讲述体验的系统性升级：从“用户能不能稳定说完一段话”，推进到“用户能看到自己说了什么、AI 提问不会重复占空间、主题讲丰富后能自然换题、语音不方便时能打字、统计数据和后台一致”。整体目标是让故事坊的主对话链路更可靠、更省空间、更符合真实老人使用场景。

## 一、长语音识别稳定性优化

最早的问题来自用户长时间说话时，系统容易出现“没听清”“识别失败”的体验问题。尤其是一分钟、两分钟这类较长语音，如果一次性送入识别服务，容易受到接口时长、音频体积、网络波动、静音段和浏览器录音稳定性的影响。

本轮处理方向：

- 后端新增长音频分片识别能力。
- 对 PCM 16k 16bit mono 音频按时长切片，并保持采样点对齐。
- 每段音频独立识别，最后合并非空文本。
- 避免单段音频过长导致整段失败。
- 继续兼容现有语音识别供应商，不破坏腾讯/豆包语音 provider 切换能力。

关键收益：

- 用户说得更长时，不再因为单次请求过大而整段丢失。
- 识别结果可以按分片逐步兜底，提升长讲述成功率。
- 后续如果接实时 ASR，也可以复用“分段、合并、容错”的思路。

## 二、用户语音转写进入聊天区

原来的聊天区主要展示 AI 的话，用户自己说了什么不够清楚。用户提出：按住说话时希望像实时语音转文字一样看到自己的内容；放桌上畅聊时，希望录完后也能看到这一段识别文字。

本轮落地：

- 新增用户语音转写事件协议。
- 前端收到用户最终转写后，生成用户气泡。
- 如果已经有草稿气泡，则用最终识别结果替换草稿。
- 如果识别失败，会在用户气泡上给出“没有听清”的状态，而不是无声失败。
- 按住说话和桌上畅聊都走统一的用户转写展示逻辑。

关键收益：

- 用户能确认系统到底听到了什么。
- 家属或管理员回看时，聊天顺序更完整。
- 对老人更友好：出错时能看见问题，不会只觉得“机器没反应”。

## 四、回访开场入库规则重构

用户明确提出：回访开场提问不应该一出现就写入 CloudBase，因为如果用户没有回答，这条 AI 消息就不应该进入正式对话素材；但一旦用户回答了，就应该把这条开场提问和用户回答形成一组问答，纳入正常聊天记录。

本轮确定并实现的规则：

- AI 回访开场只先存在于会话内存。
- 未被用户回答时，不写入 `conversations`。
- 用户回答后，把该开场提问作为本轮 AI prompt 元数据写入记录。
- 该 AI prompt 可用于聊天展示和上下文衔接。
- 同时标记为不进入摘要、不进入传记素材、不参与统计污染。

这样做的核心原因：

- 避免“未回答问题”污染回忆录素材。
- 保留真实问答上下文。
- 让回访开场既能展示，又不破坏数据库素材质量。

相关文件：

- `lib/entryGuidanceTurn.js`
- `server.js`
- `tests/entryGuidanceTurn.test.js`

## 五、AI 神态描述过滤

测试过程中发现 AI 会输出类似：

```text
（温和地笑着，眼神中充满鼓励）
```

这类内容如果被 TTS 读出来，会显得很奇怪，破坏自然交流体验。

本轮处理：

- 在 AI 回复进入展示和朗读前，清理括号中的非口语舞台描述。
- 保留正常口语内容。
- 如果 AI 回复只剩舞台描述，则回退到兜底回复。
- 约束用户听到的是“人会自然说出口的话”，不是剧本旁白。

关键收益：

- TTS 不再朗读神态动作。
- 对话更像真实陪伴，而不是表演文本。
- 老人端听感更自然。

## 六、富主题 85% 换题机制

当某个传记主题已经讲到 85% 以上，用户仍然可能一直围绕同一主题聊下去。用户提出，希望 AI 在主题足够丰富后温和询问：继续当前主题，还是换一个主题。

本轮实现：

- 当前主题进度达到 85% 后，AI 会触发富主题换题提示。
- 提示不是强制换题，而是让用户选择：
  - 继续当前主题
  - 换到下一个未充分讲述的主题
- 支持用户用自然语言回答：
  - “继续”
  - “换一个”
  - 具体主题名
  - “去回忆库看看”
- 用户选择换题后，系统会生成新主题开场问题，而不是继续沿用旧主题上下文。
- 同一主题不会在短时间内反复弹出换题提示。

关键收益：

- 避免老人长期卡在一个话题。
- 保持讲述节奏自然，不生硬打断。
- 让传记素材覆盖更均衡。

## 十、用户端统计修复：100 段上限问题

后台管理端显示郑远有 149 轮对话，但用户端长期显示 100 段。排查后确认原因是后台和用户端走的统计接口不同：

- 管理后台使用 CloudBase `.count()`，所以显示 149。
- 用户端 `/api/stats/:userId` 原先使用 `.get().data.length`，受到 CloudBase 默认分页限制，只能拿到前 100 条。

本轮修复：

- 用户端统计接口改为使用 `.count()` 获取真实总数。
- 音频时长估算仍需要读取 `audioSizeKB`，因此对 conversation 记录做分页读取。
- 分页读取只用于累计音频大小，不再用于统计条数。

关键收益：

- 用户端和后台管理端统计口径一致。
- 超过 100 条后不再显示错误数字。
- 保留音频时长估算能力。

注意事项：

- 如果本地页面仍显示 100，通常是后端 `node server.js` 旧进程没有重启。
- 重启后端后，`/api/stats/:userId` 应返回真实 `totalConversations`。

## 十一、TTS 额度消耗问题记录

今日确认了一条终端日志的含义：

```text
入口引导 TTS 音频已发送 63.0 KB
```

结论：

- `63.0 KB` 是生成后发送到前端的音频大小，不是 token 数。
- 但这条日志出现，说明后端已经调用过一次 TTS 语音合成接口。
- 它会消耗语音合成 API 额度。
- 它不消耗 ASR 语音识别额度。
- 它也不是 DeepSeek/豆包主模型文本 token，除非同一轮另外调用了大模型。

已记录的后续优化方向：

- 入口引导默认只显示文字，不自动 TTS。
- 用户点击“播放”时才生成 TTS。
- 对相同入口引导文本做音频缓存，避免刷新或重连重复消耗。
- 文本输入模式下继续保持不自动朗读。

本项今日只做问题确认与方案记录，尚未进入工程实现。

## 十四、当前遗留事项

### 14.1 TTS 额度优化

后续优先级较高。建议下一轮单独设计：

- 自动朗读策略
- 播放按钮交互
- TTS 音频缓存

用户端统计已经修复为 `.count()`，但本地页面如果还显示 100，需要重启后端：

```bash
node server.js
```

然后重新登录或刷新页面。

### 14.3 React Hook 依赖 warning

当前定向 lint 没有错误，但 `useStoryEngine.ts` 仍有 3 个 hook dependency warning。后续可以单独清理，避免在对话状态、录音停止、WebSocket 消息处理上出现隐性闭包问题。

## 今日结论

今天这轮开发把“讲故事主链路”向前推进了一大步：语音更稳、文本可用、用户自己说的话可见、AI 提问不重复占位、主题丰富后能换题、统计数字和后台一致、移动端空间更合理。

产品层面，故事坊从单纯语音聊天，升级为“语音 + 文本 + 主题节奏 + 回访上下文 + 真实数据统计”的完整讲述工作台。下一步最值得做的是 TTS 额度优化，让系统在保持陪伴感的同时，降低不必要的语音合成成本。
