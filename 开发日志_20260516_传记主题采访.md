## 二、产品决策记录

### 2.1 右侧栏功能取舍

讨论后决定，第一版右侧栏不再承担“亲情关怀、紧急求助、分享给家人、传记准备进度”等横向功能。

原因：

- “紧急求助”会把产品预期拉向老人安全监护，带来责任边界问题。
- “亲情关怀”和“分享给家人”更像后续家庭协作能力，不是第一版核心闭环。
- “传记准备进度”过于结果导向，而当前更需要帮助老人开始讲述。

最终右侧栏定位调整为：

> 今天想聊哪个主题？让老人或陪护者选择一个人生主题，由 AI 围绕该主题进行采访。

### 2.2 第一版 10 个主题

最终确定的第一版主题如下：

1. 我的孩童时代
2. 我的父母和家
3. 求学时候的日子
4. 年轻时候的日子
5. 工作与生计
6. 爱情与婚姻
7. 家庭与子女
8. 人生的转折点
9. 难忘的年代
10. 留给家人的话

这些主题只是“采访入口”，不是最终成书章节标题。后续写作章节标题会根据实际素材自动提炼。

### 2.3 进度规则

每个主题的进度不是简单按聊天时长计算，而是按“素材丰富度”判断。

统一评分维度：

| 维度 | 分值 | 含义 |
|------|------|------|
| 基础事实 | 20 | 时间、地点、人物、身份关系是否清楚 |
| 具体事件 | 20 | 是否讲出真实发生过的故事 |
| 细节画面 | 20 | 是否有生活细节、场景、物件、声音等 |
| 情绪感受 | 20 | 是否表达开心、害怕、遗憾、自豪、想念等 |
| 人生影响 | 20 | 是否说明这段经历如何影响后来的人生 |

状态分桶：

| 进度 | 状态文案 |
|------|----------|
| 0% | 还没开始 |
| 1-34% | 刚刚开始 |
| 35-64% | 已有故事 |
| 65-84% | 继续补充 |
| 85%+ | 素材已丰富 |

重要产品原则：

> 达到 85% 后，只提示“这个主题素材已经很丰富”，不强制切换主题，老人仍然可以继续讲。

## 三、数据库与数据结构

### 3.1 新增集合

新增腾讯云 CloudBase 集合：

```text
topic_profiles
```

设计原则：

- 不改动旧集合结构。
- 不重命名旧字段。
- 不改变 `memory_profiles` 的语义。
- 主题采访进度独立存储，避免影响已有自传生成链路。

### 3.2 `topic_profiles` 文档结构

每个用户对应一条主题档案：

```js
{
  userId: string,
  currentTopicId: "childhood",
  personProfile: {},
  topics: [
    {
      id: "childhood",
      title: "我的孩童时代",
      progress: 0,
      status: "not_started",
      summary: "",
      knownFacts: [],
      concreteStories: [],
      missingInfo: [],
      suggestedNextQuestion: "",
      lastDiscussedAt: null
    }
  ],
  allRichPromptShown: false,
  createdAt,
  updatedAt
}
```

### 3.3 当前已完成的数据能力

后端已支持：

- 登录/注册后自动获取或创建 `topic_profiles`。
- 当前主题默认是 `childhood`。
- 可通过 HTTP API 获取用户主题档案。
- 可通过 HTTP API 或 WebSocket 切换当前主题。
- 如果数据库中已有旧版本主题档案，后端会补齐缺失字段。

## 十二、后续补充：退出登录时停止浏览器语音播放

新增和调整文件：

```text
lovable_ui/src/lib/audioSessionGuard.js
lovable_ui/src/lib/audioSessionGuard.d.ts
tests/audioSessionGuard.test.js
lovable_ui/src/hooks/useStoryEngine.ts
```

问题：

AI 正在朗读时，用户点击退出登录进入登录页，但浏览器里的语音仍继续播放。

根因：

- 退出登录时只关闭 WebSocket 和停止录音。
- 没有停止当前 `AudioBufferSourceNode`。
- WebSocket 关闭是异步的，退出后短时间内可能仍收到迟到音频包。

修复：

- `logout()` 中调用 `stopPlayback()`，立即停止当前音频并清空队列。
- 退出时清空字幕、录音状态和音频可视化数据。
- 新增前端音频 session guard。
- 退出后如果收到迟到音频包，前端直接丢弃，不再进入播放队列。
- 组件卸载或用户切换时也执行播放清理。

## 十三、后续补充：AI整理成果页与继续讲讲

### 13.1 产品定位

将左侧 `AI整理` 从“功能建设中”改为老人端的“回忆整理小报”。

设计原则：

- 老人端不展示复杂后台数据。

### 13.2 页面结构

`AI整理` 页面分为 4 块：

1. **今天整理**

2. **故事片段**
   - 展示 AI 已整理出的可读故事素材。
   - 每条只显示标题和简短摘要。

3. **重要的人和地方**
   - 展示人物和地点标签。
   - 例如：母亲、父亲、河南老家、纺织厂。

4. **原始记录**
   - 作为底部弱入口。
   - 只展示最近几条预览，不默认铺满完整聊天日志。

### 13.4 继续讲讲推荐逻辑

产品规则：

> “继续讲讲”只来自最近真实聊过的内容，不从没聊过的主题里凭空推荐。

推荐对象结构：

```js
{
  type: "continue_recent",
  title: "继续讲讲刚才的故事",
  question: "小时候跟母亲去河边，您印象最深的是哪一次？",
  topicId: "childhood",
  topicTitle: "我的孩童时代",
  sourceType: "recent_summary",
  sourceId: "summary_xxx"

```

主题归属原则：

- 推荐生成时就绑定 `topicId`。
- 点击时不再用关键词猜主题。
- 点击时只执行推荐对象里已有的 `topicId`。

### 13.5 点击继续讲讲后的正式记录与朗读

新增文件：

```text
lib/recommendationQuestion.js
tests/recommendationQuestion.test.js
lovable_ui/src/lib/archiveTypes.ts
```

新增 WebSocket 事件：

```js
{
  type: "start_recommendation_question",
  topicId,
  question,
  title,
  sourceType,
  sourceId
}
```

后端处理流程：

1. 校验用户已登录。
2. 校验推荐问题和主题。
3. 切换当前主题。
4. 将推荐问题保存为正式 AI 提问记录。
5. 向前端发送 `ai_speaking` 文本事件。
6. 使用现有 TTS 合成推荐问题语音。
7. 前端播放推荐问题。
8. 当前 session 临时保存 `pendingRecommendationQuestion`。
9. 老人回答后，下一轮 AI prompt 注入“上一句 AI 推荐追问”。
10. 用户回答处理完成后清除 pending question。

推荐问题保存到 `conversations` 时使用可选字段：

```js
{
  userText: "",
  aiReply: question,
  topicId,
  topicTitle,
  topicProgress,
  source: "archive_recommendation",
  recommendation: {
    title,
    sourceType,
    sourceId
  }
}
```

这样管理员端和原始记录中可以看到：

- 这是一次由 AI整理推荐触发的正式 AI 提问。
- 老人之后的回答仍沿用原有 ASR、保存对话、AI 回复、TTS、主题分析流程。
