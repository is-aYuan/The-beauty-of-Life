## 一、今日工作概览

今日完成了故事坊从"能对话"到"能写书"的关键跨越。核心工作围绕三个主题展开：

1. **叙事摘要提取系统** — 每次对话后自动提取结构化摘要
2. **管理员后台** — 用户数据管理和系统监控
3. **自传生成系统** — 从对话到成书的完整链路设计与实现

## 二、核心讨论与设计决策

### 2.1 长对话上下文问题

**问题提出：** 当用户对话积累过多后，用 AI 生成自传时会遇到上下文窗口溢出、识别错误、内容不足或幻觉等问题。

**讨论过程：**

最初提出了三层架构方案：
- 第1层：原始对话（完整保留，永不删除）
- 第2层：叙事摘要（保留原话、情感、细节的结构化数据）
- 第3层：自传生成（输出层）

**关键争论：** "干净的素材是否等于内容含量极少？"

结论是摘要不是"一句话总结"，而是"叙事级摘要"——保留用户原话引用、情感状态、具体生活细节的完整段落。100轮对话压缩成10段叙事摘要，字数只有原来的1/5，但信息密度足够写自传。

### 2.2 AI 提取方案选择

**讨论：** 用混元还是 DeepSeek 进行摘要提取？

最终选择 DeepSeek（deepseek-chat/V3），理由：
- 混元用于对话交互，DeepSeek 用于内容提取，职责分离
- DeepSeek 的 JSON 输出稳定性更好
- 成本可控

### 2.3 自传生成的核心设计讨论

#### 问题一：什么时候算"素材足够"？

最初提出清单式检查（童年✅ 工作❌），但这种方式太僵硬——用户不会按清单讲故事。

最终方案：**五个维度 + 素材量**

| 维度 | 判断标准 |
|------|----------|
| timeline | 能否从出生到现在大致排出来 |
| keyPeople | 是否提到了父母、配偶、子女中的两类 |
| depth | 是否有某个话题讲了3轮以上 |
| stories | 是否有带场景、对话的完整故事 |
| emotions | 是否表达过内心感受、反思、遗憾 |

就绪条件：5维度 ≥ 4个达标 + 素材 ≥ 10000字

#### 问题二：用户思维跳跃怎么办？

**关键转变：** 从"控制用户"到"跟随用户"。

最初方案是清单驱动（"您该讲求学经历了"），但这种方式：
- 假设人生话题是离散、可序化的
- 给用户造成压力
- 像在填表，不是在聊天

最终方案：**AI 跟随用户，记忆档案累积**

- 用户想聊什么就聊什么
- AI 像一个认真听故事的人，默默记住所有信息
- 下次聊天时自然引用已知信息（"您上次说那年冬天特别冷……"）
- 不是生硬拉回话题，而是自然关联

#### 问题三：AI 判断"足够"的标准是什么？

不能靠 AI 的感觉，需要可衡量的条件。将"足够"拆成五个可检查的维度，每个维度有明确的 true/false 判断标准。

#### 问题四：产出档次如何确定？

**讨论：** 是否让用户选择写多少字？

结论：**不让用户选择，系统自适应。** 对老年人来说，少做选择就是最好的设计。

| 素材量 | 产出档次 | 章节数 | 每章字数 |
|--------|----------|--------|----------|
| < 10,000字 | 人生故事集 | 5-8章 | 800-1500字 |
| 10,000-30,000字 | 个人传记 | 8-12章 | 1500-2500字 |
| > 30,000字 | 完整回忆录 | 12-20章 | 2000-3000字 |

用户无感，系统根据素材量自动选择最合适的档次。

#### 问题五：生成自传由谁触发？

**讨论：** 管理员生成还是用户自己生成？

结论：**用户主动触发为主，系统检测提示为辅。**

两条路径：
1. 系统检测达标 → AI 登录时自然提示："您讲的故事已经很完整了，要我帮您整理成书吗？"
2. 用户随时可以点"生成我的故事书"按钮

管理员只负责查看和管理，不参与生成决策。

#### 问题六：自传素材存在哪里？

**讨论：** 自传素材（biography-material）是否需要持久化？

结论：自传素材是 API 临时拼接的，不存储。真正需要存储的是：
- 叙事摘要（summaries 集合）— 每次会话后自动产生
- 记忆档案（memory_profiles 集合）— 累积更新
- 成品自传（biographies 集合）— 生成后存储

### 2.4 管理员系统设计

**讨论：** 是否需要管理员系统？

方案选择：方案A（简单管理员），理由：
- 内测阶段只有一个人需要管理
- 改动最小，半天能做完
- 正式上线时可升级到角色权限系统

管理员功能：
- 系统统计（用户数、会话数、对话数、摘要数）
- 用户列表（含每个用户的统计数据）
- 用户详情（对话记录、叙事摘要、记忆档案、成品自传）
- 删除用户（级联清理全部6个集合）
- 触发生成自传

## 三、技术实现

### 3.1 数据库结构（CloudBase）

共6个集合：

| 集合名 | 用途 | 关联字段 |
|--------|------|----------|
| `users` | 用户资料 | - |
| `sessions` | 会话记录 | userId |
| `conversations` | 对话记录 | userId, sessionId |
| `summaries` | 叙事摘要 | userId, sessionId |
| `memory_profiles` | 记忆档案 | userId |
| `biographies` | 成品自传 | userId |

### 3.2 完整数据链路

```
用户语音对话
    ↓
WebSocket 接收音频
    ↓
腾讯云 ASR 语音识别
    ↓
混元大模型 AI 对话
    ↓
腾讯云 TTS 语音合成
    ↓
会话结束（WebSocket 断开）
    ↓
extractNarrativeSummary()
    ├→ 调用 DeepSeek 提取叙事摘要 + 记忆档案
    ├→ 存入 summaries 集合
    └→ updateMemoryProfile() → 累积更新 memory_profiles 集合
         └→ isReady() 判断就绪度

用户下次登录
    ↓
getWelcomeText()
    ├→ 有 biographies → "您的故事书写好了！"
    ├→ 就绪 → "素材够了，要生成吗？"
    └→ 未就绪 → "请继续讲故事。"

用户点击"生成我的故事书"
    ↓
generateBiography()
    ├→ 拉取所有 summaries + memory_profiles
    ├→ 根据素材量确定产出档次
    ├→ 调用 DeepSeek 撰写自传
    └→ 存入 biographies 集合
```

### 3.3 Prompt 设计

#### SUMMARY_SYSTEM_PROMPT（叙事摘要 + 记忆档案提取）

职责：每次对话结束后调用，提取结构化数据。

输出字段：
- `profile` — 用户基本信息（姓名、年龄、籍贯）
- `narratives` — 叙事段落数组（主题、标题、内容、关键事实）
- `coverage` — 话题覆盖情况（已讨论、未涉及）
- `emotionalNote` — 情感状态描述
- `memoryArchive` — 记忆档案（人物、地点、事件、情感）
- `readiness` — 就绪度判断（5个维度各自的状态和理由）

关键规则：
1. 用户原话必须用引号完整保留，不得改写
2. 情感状态用括号标注在叙事中
3. 保留所有具体细节（人名、地名、年份、数字）
4. content 字段必须是至少3句话的完整叙事段落
5. 就绪度每个维度必须给出判断理由

#### BIOGRAPHY_SYSTEM_PROMPT（自传撰写）

职责：用户触发生成时调用一次，撰写完整自传。

输出字段：
- `title` — 自传标题（从故事中提炼，不超过15字）
- `chapters` — 章节数组（序号、标题、正文）

关键规则：
1. 不编造事实，素材不足时留白
2. 保留用户原话引用
3. 保留不确定性（"大概是六几年吧"不替用户确定）
4. 语言温暖朴实，不要华丽辞藻
5. 有场景感、情感层次、节奏感
6. 第一章开头有吸引力，最后一章有收束感
7. 书名从用户故事中提炼（好的：《铁路上的三十年》；不好的：《张大爷自传》）

## 五、设计原则总结

### 5.1 适老化设计

- 大字号（状态文字 2.4rem，按钮 1.8rem）
- 大触控目标（最小 48px，麦克风按钮 180px）
- 高对比度（白色文字，深色背景）
- 清晰反馈（录音脉冲动画、状态颜色变化）
- 减少选择（不问用户选哪个档位，系统自适应）

### 5.2 数据安全

- 用户数据通过 userId 隔离
- 管理员需要 token 认证（2小时过期）
- 删除用户级联清理全部6个集合
- .env 不提交到代码仓库

### 5.3 工程化思维

- 错误处理：每个 API 端点都有 try-catch
- 异步处理：摘要提取不阻塞会话关闭
- 向后兼容：新增字段有默认值
- 职责分离：混元管对话，DeepSeek 管提取和撰写
- 日志追踪：每个关键步骤都有 console.log
